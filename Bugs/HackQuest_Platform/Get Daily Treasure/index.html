🎁 Bug Report

"Get Daily Treasure" Button Triggers Client-Side Error After Completing All Requirements

🔴 Severity: High 🌐 Platform: Web 📅 Date: May 2026 🎯 Feature: Daily Rewards

Bug Summary

⚠️ "Get Daily Treasure" Reward Button Fails with Generic Client Error

On the HackQuest home page, after a user successfully completes all three daily requirements (claim daily streak, like 4 projects, finish 1 unit), the "Get Daily Treasure" reward button becomes available. However, clicking this button triggers a generic client-side error message: "Application error: a client side error has occurred (see browser console for more info)". Investigation of the browser console reveals no actionable error details, preventing users from claiming earned rewards and causing frustration.

🔴 Severity: High 💸 Impact: Revenue & Retention

Steps to Reproduce

  1. Access HackQuest Home Page: Navigate to https://www.hackquest.io and log in to your account
  2. Complete Daily Requirement #1: Claim your daily streak points from the streak widget
  3. Complete Daily Requirement #2: Like exactly 4 different projects in the community feed
  4. Complete Daily Requirement #3: Finish learning 1 complete unit in any course
  5. Observe Reward Activation: Confirm the "Get Daily Treasure" button appears and is clickable
  6. Trigger the Bug: Click the "Get Daily Treasure" button
  7. Verify Error: Note the error modal: "Application error: a client side error has occurred"
  8. Check Console: Open browser DevTools → Console tab and observe lack of meaningful error details

Expected vs Actual Behavior

✅ Expected Behavior

After fulfilling all three daily conditions, clicking "Get Daily Treasure" should successfully process the reward claim, display a confirmation animation/modal, update the user's inventory or points balance, and mark the daily treasure as claimed for that day.

❌ Actual Behavior

Clicking the button immediately triggers a generic error modal with the message "Application error: a client side error has occurred (see browser console for more info)". The browser console shows no useful stack trace, network errors, or validation messages. The reward is not granted, and the button remains in a broken state.

Potential Root Causes

🔍 Technical Investigation Points:

  • Missing Error Handling: The reward claim API call may lack proper try/catch or error boundary handling, causing uncaught promises to bubble up as generic errors
  • Silent API Failure: Backend endpoint for claiming treasure might return a 5xx error or malformed response that the frontend doesn't gracefully handle
  • State Validation Gap: Frontend may not re-validate all three conditions client-side before submitting, causing race conditions if server state differs
  • Authentication Token Issue: The claim request might fail due to expired/missing auth tokens that aren't refreshed before the API call
  • CORS or Network Interception: Browser extensions, ad-blockers, or strict privacy settings could be intercepting the request without clear feedback
  • Missing Error Logging: Application may swallow errors intentionally for UX but fail to log them to monitoring services (Sentry, LogRocket, etc.)
  • Dependency Version Conflict: Recent updates to reward logic dependencies (e.g., web3 libraries, state management) may introduce silent failures

Recommended Solutions

  1. Implement Structured Error Handling: Wrap the treasure claim function in comprehensive try/catch blocks with specific error messages for different failure modes
  2. Add Client-Side Pre-Validation: Before submitting the claim, re-check all three conditions against the latest user state to prevent invalid requests
  3. Enhance Error Logging: Integrate with error monitoring tools to capture full stack traces, user context, and request payloads when errors occur
  4. Improve User Feedback: Replace generic error modals with actionable messages (e.g., "Reward claim failed. Please try again or contact support if the issue persists")
  5. Add Retry Logic: Implement exponential backoff retry for transient network failures with clear loading states
  6. Backend Health Check: Add a pre-flight endpoint to verify reward system availability before enabling the claim button
  7. QA Test Coverage: Add E2E tests covering the full daily reward flow, including edge cases like partial completion and network interruptions

Additional Notes

⚠️ Business Impact: This bug directly affects user retention and engagement. Daily rewards are a core gamification mechanic—failure to claim earned rewards erodes trust and may cause users to abandon daily participation. Additionally, if the treasure includes tokens or NFTs, this represents a potential revenue loss. The lack of console visibility significantly increases debugging time and delays resolution.

💡 Quick Mitigation: As a temporary workaround, consider adding a "Report Issue" button in the error modal that auto-collects user context and sends a structured report to support/engineering.
